# 5 系统实现

系统实现是指将软件系统的设计思想转化为实际代码的过程。在系统实现阶段，开发人员需要将设计文档转化为实际代码，以实现系统的各种功能和特性。根据用户需求分析、系统设计进一步编写系统代码。本章将详细介绍 RLMusic 本地音乐播放器各核心模块的实现过程与关键技术细节。

## 5.1 用户登录注册

用户访问本系统时，首先需要进行身份认证。如图 5.1 是系统的登录注册流程图。新用户需要注册一个账户，输入用户名、密码和邮箱后提交注册请求。系统会在数据库中检查用户名是否已存在，若存在则提示用户更换用户名；若不存在则使用 Bcrypt 算法对密码进行加密后存入数据库，并为用户自动创建默认的"我喜欢的音乐"私有歌单。注册完成后返回登录页面，用户输入已注册的账户和密码，点击登录后系统进行身份认证，验证通过后会生成 JWT Token 返回给前端，并跳转到系统首页。

当用户输入正确的用户名和密码并点击提交按钮时，前端通过 Axios 将数据发送至后端 `/auth/login` 接口。后端使用 Gin 框架接收请求，如代码 5-1 所示。首先根据用户名查询数据库，若用户不存在则返回相应错误；若存在则使用 `encrypt.BcryptCheck` 方法比对密码。验证通过后调用 `jwt.GenToken` 生成 JWT 令牌，并更新用户的最后登录时间与 IP 地址，最终返回包含用户信息和 Token 的响应。

代码 5-1 后端用户登录实现
```go
func (*UserAuth) Login(c *gin.Context) {
    var req LoginReq
    if err := c.ShouldBindJSON(&req); err != nil {
        ReturnError(c, g.ErrRequest, err)
        return
    }

    db := GetDB(c)
    user, err := model.GetUserAuthInfoByName(db, req.Username)
    if err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            ReturnError(c, g.ErrUserNotExist, nil)
            return
        }
        ReturnError(c, g.ErrDbOp, err)
        return
    }

    if !encrypt.BcryptCheck(req.Password, user.Password) {
        ReturnError(c, g.ErrPassword, nil)
        return
    }

    conf := g.Conf.JWT
    token, err := jwt.GenToken(conf.Secret, conf.Issuer, int(conf.Expire), user.ID)
    if err != nil {
        ReturnError(c, g.ErrTokenCreate, err)
        return
    }

    model.UpdateUserLoginInfo(db, user.ID, c.ClientIP())
    ReturnSuccess(c, LoginVO{
        ID: user.ID, Username: user.Username,
        Email: user.Email, Avatar: user.Avatar,
        UserGroup: user.UserGroup, Token: token,
    })
}
```

注册页面与登录页面共用同一组件，通过 `n-tabs` 标签页进行切换，如图 5.3 所示。用户输入用户名、密码和邮箱后，前端调用 `/auth/register` 接口。后端首先校验密码长度，再检查用户名是否已被注册。若用户名因软删除而被占用，则重命名旧记录以释放用户名。最后使用 Bcrypt 加密密码并将用户信息写入数据库，同时创建默认私有歌单。

身份验证功能采用 JWT（JSON Web Token）无状态认证方案实现。在用户浏览器与系统服务器通信过程中，后端通过 `JWTAuth` 中间件拦截请求，从 HTTP Header 的 `Authorization` 字段中提取 Token，解析并校验 Token 的有效期和签名。如代码 5-2 所示，中间件解析 Token 后根据 `UserId` 查询数据库获取完整的用户信息，并将其注入到 Gin 的 Context 中，供后续业务处理器使用。

代码 5-2 JWT 认证中间件实现
```go
func JWTAuth() gin.HandlerFunc {
    return func(c *gin.Context) {
        db := c.MustGet(g.CtxDB).(*gorm.DB)
        authorization := c.Request.Header.Get("Authorization")
        if authorization == "" {
            handle.ReturnError(c, g.ErrTokenNotExist, nil)
            return
        }

        parts := strings.Split(authorization, " ")
        if len(parts) != 2 || parts[0] != "Bearer" {
            handle.ReturnError(c, g.ErrTokenType, nil)
            return
        }

        claims, err := jwt.ParseToken(g.Conf.JWT.Secret, parts[1])
        if err != nil {
            handle.ReturnError(c, g.ErrTokenWrong, err)
            return
        }

        if time.Now().Unix() > claims.ExpiresAt.Unix() {
            handle.ReturnError(c, g.ErrTokenRuntime, nil)
            return
        }

        user, err := model.GetUserAuthInfoById(db, claims.UserId)
        if err != nil {
            handle.ReturnError(c, g.ErrUserNotExist, err)
            return
        }
        c.Set(g.CtxUserAuth, user)
    }
}
```

前端在登录成功后将 Token 保存到 `sessionStorage` 中，并在后续请求的 Axios 拦截器中自动附加到 Header。同时，前端支持"记住密码"功能，使用 AES 对称加密算法将用户名和密码安全地存储在 `localStorage` 中，下次访问时自动回填。

## 5.2 导航栏

### 5.2.1 全局布局框架

系统采用顶部常驻导航栏设计，作为所有非全屏页面的全局入口。页面整体布局由 `src/AppContent.vue` 负责，使用 Naive UI 的 `n-layout`、`n-layout-header` 和 `n-layout-content` 组件构建基础框架。如代码 5-3 所示，对于登录页、初始化配置页等需要全屏展示的路由，通过 `route.meta.hideLayout` 控制是否隐藏导航与播放器，从而实现沉浸式体验；普通页面则在 `n-layout-header` 中挂载 `Nav` 组件，在布局底部预留播放器空间。

代码 5-3 全局页面布局与路由适配
```vue
<template>

    >
      <main ref="mainContent" class="main-container">

          :bottom="music.getPlaylists[0] && music.showPlayBar ? 120 : 40"
```

`AppContent.vue` 还负责多项全局交互逻辑：监听空格键实现播放/暂停快捷控制、服务器连接失败时弹出配置模态框、基于 `localStorage` 中 AES 加密凭据的自动登录等。这些功能与导航状态紧密关联，确保用户无论在哪个页面都能获得一致的操作体验。

导航栏右侧集成了用户头像下拉菜单与管理员入口，其显示逻辑与用户登录状态深度绑定。用户头像点击时，若未登录则直接跳转至 `/login`；已登录则展开下拉菜单。下拉菜单内容通过 `computed` 动态计算，如代码 5-5 所示，包含用户信息卡片渲染项、全局设置、登录/退出切换以及关于本站等选项。其中用户信息卡片采用 Naive UI 的 `render` 类型选项，通过 Vue 的 `h()` 函数手动构建虚拟 DOM，展示用户头像、昵称和邮箱，点击后可进入个人中心。

管理员入口在点击时会进行前端权限校验。系统从 Pinia 用户状态 store 中读取当前登录用户的 `userGroup` 字段，仅当值为 `admin` 时才允许跳转至管理员后台 `/admin`，否则通过 Naive UI 的 `message.warning` 提示"当前账号非管理员"。这种前端校验不仅提升了用户体验，后端在所有管理员接口中也会进行二次权限验证，形成双重保障。

### 5.2.4 响应式适配与移动端菜单

导航栏具备良好的响应式适配能力。通过 CSS 媒体查询 `@media (max-width: 768px)`，桌面端显示完整的链接与图标按钮；在平板或手机等窄屏设备下，中间导航链接、前进/后退按钮和部分功能图标自动隐藏，取而代之的是右侧的汉堡菜单按钮。点击汉堡菜单后，通过 Naive UI 的 `n-drawer` 从右侧滑出宽度为 280px 的抽屉式菜单，供用户进行页面切换。

移动端抽屉菜单复用了与桌面端相同的导航逻辑，点击任意菜单项后自动关闭抽屉并触发路由跳转。这种"移动端降级"策略在保证小屏幕可用性的同时，避免了导航栏元素过度拥挤。此外，`AppContent.vue` 中的主内容区设置了 `max-width: 1440px` 并自动居中，确保在超宽屏显示器上内容不会过度拉伸，维持舒适的阅读与操作区域。

`AppContent.vue` 中还配置了路由跳转的过渡动画 `fade-slide`：页面进入时从下方淡入，离开时向上淡出，配合 `mode="out-in"` 确保不会出现两个页面同时存在的闪烁问题。这些细节共同构成了导航系统流畅、连贯的用户体验。

导航栏将用户引导至系统的四大核心页面：发现页、搜索页、系统信息页和个人中心。如图 5.5 所示，发现页（首页）位于 `src/views/Home/index.vue`，采用温暖米色调背景与动态径向渐变装饰，顶部显示大号欢迎语“Hello, [昵称]”。页面中部设有“局域网访问”功能卡片，以渐变色背景展示当前服务器局域网 IP 地址与二维码。当设备存在多个网络接口（如同时连接 Wi-Fi 与以太网，或安装了虚拟网卡）时，系统会自动枚举所有可用局域网 IP，默认展示最常用的内网地址。用户点击卡片中的 IP 地址或二维码即可展开详情弹窗，弹窗内提供网卡切换列表，点击不同网卡项可实时切换对应的 IP 与二维码，方便位于不同局域网内的设备找到正确的访问入口；同时支持一键复制访问链接到剪贴板，便于直接分享。局域网内用户无需安装任何 App，只需使用手机、平板等设备的浏览器扫描二维码，即可直接访问系统 Web 页面，实现跨终端的无缝音乐共享。如图 5.5 所示。

代码 5-4 播放模式切换逻辑
```typescript
setPlaySongIndex(value: "next" | "prev" | number) {
  if (typeof value === "number") {
    this.persistData.playSongIndex = value;
  } else {
    const len = this.persistData.playlists.length;
    if (len === 0) return;

    if (this.persistData.playSongMode === "random") {
      if (len === 1) return;
      let newIndex = this.persistData.playSongIndex;
      while (newIndex === this.persistData.playSongIndex) {
        newIndex = Math.floor(Math.random() * len);
      }
      this.persistData.playSongIndex = newIndex;
      return;
    }

    if (value === "next") {
      this.persistData.playSongIndex = (this.persistData.playSongIndex + 1) % len;
    } else {
      this.persistData.playSongIndex = (this.persistData.playSongIndex - 1 + len) % len;
    }
  }
}
```

歌词同步功能通过解析 LRC 格式歌词实现。后端在 `/song/lyric/:id` 接口中，首先尝试读取与歌曲文件同名的 `.lrc` 歌词文件及 `.fy.lrc` 翻译文件；若不存在，则尝试从音频文件的 ID3 标签中读取内嵌歌词。前端获取歌词后，使用 `lyricFormat.js` 将其解析为带时间戳的数组。在 `Player` 组件中，`timeupdate` 事件触发时根据当前播放时间计算匹配的歌词索引，实现歌词高亮与滚动。当无歌词时，界面显示"暂无歌词"的默认提示。

音量控制通过 `n-slider` 垂直滑块组件实现，支持 0~1 的连续调节。点击静音图标时，调用 `volumeMute` 方法将当前音量保存到 `playVolumeMute` 中，再将音量设为 0；再次点击则恢复之前的音量值。播放器还支持音乐渐入渐出效果，在播放开始时通过 `setInterval` 逐步增加音量，在暂停时逐步降低音量，提升用户体验。

状态持久化功能借助 `pinia-plugin-persistedstate` 插件实现。用户的播放列表、当前歌曲索引、播放进度、音量、播放模式等数据会自动同步到 `localStorage` 中。页面刷新后，Store 会自动从 `localStorage` 恢复状态，播放器组件在 `onMounted` 钩子中读取上次的播放进度并设置到 Audio 元素上，从而实现无缝恢复。

此外，系统还支持"播客模式"。当用户在设置中开启播客模式且当前歌曲存在 AI 生成的开场白音频时，播放器会优先播放开场白音频（`opening_audio_file`），开场白结束后再自动切换到正文歌曲播放，实现类电台节目的收听体验。

歌单管理是本系统的重要功能模块，涵盖公共歌单、私有歌单、收藏歌单和"我喜欢的音乐"（红心歌单）四类。后端通过 `server/internal/handle/handle_song.go` 中的一系列接口实现歌单的增删改查与权限控制。

代码 5-5 切换歌曲喜欢状态
```go
func (*SongAuth) ToggleLike(c *gin.Context) {
    songIDStr := c.Param("id")
    user := GetCurrentUser(c)
    db := GetDB(c)

    var playlist model.Playlist
    err := db.Where("owner_id = ? AND title = ?", user.ID, "我喜欢的音乐").First(&playlist).Error
    if err != nil {
        if err == gorm.ErrRecordNotFound {
            pl, _ := model.CreatePlaylist(db, user.ID, "我喜欢的音乐", "因为热爱，所以收藏", false)
            playlist = *pl
        }
    }

    song, _ := model.GetSongByID(db, songIDStr)
    if model.IsSongInPlaylist(db, playlist.ID, song.ID) {
        model.RemoveSongFromPlaylist(db, &playlist, song)
        ReturnSuccess(c, gin.H{"liked": false, "message": "已取消喜欢"})
    } else {
        model.AddSongToPlaylist(db, &playlist, song)
        ReturnSuccess(c, gin.H{"liked": true, "message": "已添加到喜欢"})
    }
}
```

```typescript
export const getSearchSuggest = async (keywords: string) => {
    const [songs, artists, albums, playlists] = await Promise.all([
        getSearchSongs(keywords, 10),
        getSearchArtists(keywords, 5),
        getSearchAlbums(keywords, 5),
        getSearchPlaylists(keywords, 5)
    ]);

    const rawSongs = (songs as any).data?.result?.songs || [];
    const processedSongs = rawSongs.map((song: any) => ({
        ...song,
        id: song.ID || song.id,
        album_title: song.album_name || song.album_title,
        cover_url: song.cover_url || (song.id ? getSongCover(song.id) : undefined),
    }));

    const rawArtists = (artists as any).data?.result?.artists || [];
    const processedArtists = rawArtists.map((artist: any) => ({
        ...artist,
        id: artist.ID || artist.id,
        picUrl: artist.picUrl || artist.img1v1Url || resolveCoverUrl(artist.cover),
    }));

    const rawAlbums = (albums as any).data?.result?.albums || [];
    const processedAlbums = rawAlbums.map((album: any) => ({
        ...album,
        id: album.ID || album.id,
        name: album.name || album.title,
        picUrl: album.picUrl || resolveCoverUrl(album.cover),
    }));

    const rawPlaylists = (playlists as any).data?.result?.playlists || [];
    const processedPlaylists = rawPlaylists.map((playlist: any) => ({
        ...playlist,
        id: playlist.ID || playlist.id,
        name: playlist.name || playlist.title,
        coverImgUrl: playlist.coverImgUrl || playlist.picUrl || resolveCoverUrl(playlist.cover_url),
    }));

    return {
        songs: processedSongs,
        artists: processedArtists,
        albums: processedAlbums,
        playlists: processedPlaylists
    };
}
```

## 5.6 AI 智能模块

系统集成了大语言模型（LLM）和语音合成（TTS）能力，为歌单、艺术家、专辑生成智能文案描述，以及为歌曲生成播客风格的开场白。AI 功能的相关提示词模板存储在 `server/prompts/` 目录下，后端通过调用通义千问（Qwen）API 实现文本生成。

后端统一封装了 AI 调用客户端 `server/internal/utils/ai/SiliconFlow_Qwen.go`，如代码 5-7 所示。该模块使用标准的 OpenAI SDK 对接 SiliconFlow 平台，配置了基于令牌桶的速率限制器（`rate.Limiter`），将 RPM 限制转换为每秒请求速率，避免触发上游接口的限流策略。调用时会自动记录 Token 消耗情况，便于后续成本分析。

代码 5-7 SiliconFlow Qwen 对话封装与限流
```go
func ChatWithQwen(prompt string) (string, error) {
    conf := g.GetConfig()
    apiKey := conf.SiliconFlow.ApiKey
    if apiKey == "" {
        apiKey = os.Getenv("SILICON_FLOW_API_KEY")
    }
    if apiKey == "" {
        return "", fmt.Errorf("SiliconFlow API Key not found")
    }

    config := openai.DefaultConfig(apiKey)
    config.BaseURL = "https://api.siliconflow.cn/v1"
    client := openai.NewClientWithConfig(config)

    modelName := conf.SiliconFlow.Model
    if modelName == "" {
        modelName = "Qwen/Qwen3-8B"
    }

    ctx := context.Background()
    if err := limiter.Wait(ctx); err != nil {
        return "", fmt.Errorf("rate limiter wait failed: %v", err)
    }

    resp, err := client.CreateChatCompletion(
        ctx,
        openai.ChatCompletionRequest{
            Model: modelName,
            Messages: []openai.ChatCompletionMessage{
                { Role: openai.ChatMessageRoleUser, Content: prompt },
            },
            MaxTokens: 512,
        },
    )
    if err != nil {
        return "", fmt.Errorf("api call failed: %v", err)
    }
    if len(resp.Choices) > 0 {
        return resp.Choices[0].Message.Content, nil
    }
    return "", fmt.Errorf("no response content")
}
```

以歌单 AI 描述为例，后端采用两阶段生成策略：第一阶段读取 `prompt_playlist1.md`，将歌单中的歌曲数据（标题、艺术家、专辑、年份）以 JSON 格式传入，请求模型生成结构化的分析摘要；第二阶段读取 `prompt_playlist2.md`，将第一阶段的摘要作为上下文传入，请求模型生成一段优美、富有情感的歌单描述文本。核心实现如代码 5-8 所示，最终描述会保存到数据库中，并在歌单详情页展示。

代码 5-8 歌单 AI 描述两阶段生成核心逻辑
```go
func (h *SongAuth) generateAndSaveDescription(db *gorm.DB, idStr string, user *model.User) error {
    playlistDetail, err := model.GetPlaylistRandomSongs(db, idStr, 100)
    if err != nil {
        return err
    }
    // 权限检查 ...

    promptBytes, _ := os.ReadFile("prompts/prompt_playlist1.md")
    promptTemplate1 := string(promptBytes)

    type AISongData struct {
        Title  string `json:"title"`
        Artist string `json:"artist"`
        Album  string `json:"album"`

    var aiSongs []AISongData

        })

    jsonData, _ := json.Marshal(aiSongs)

    analysisReply, err := ai.ChatWithQwen(finalPrompt1)
    if err != nil {
        return err
    }

    promptBytes2, _ := os.ReadFile("prompts/prompt_playlist2.md")
    promptTemplate2 := string(promptBytes2)
    finalPrompt2 := strings.Replace(promptTemplate2, "{{summary_json}}", analysisReply, 1)
    description, err := ai.ChatWithQwen(finalPrompt2)
    if err != nil {
        return err
    }

    return db.Model(&model.Playlist{}).Where("id = ?", idStr).Update("description", description).Error
}
```

播客开场白功能采用四阶段流水线：第一步分析歌曲元数据生成摘要；第二步基于摘要生成开场白草稿；第三步将草稿润色为适合朗读的最终脚本；第四步调用 Qwen TTS 接口将脚本合成为音频文件（FLAC 格式，24kHz 采样率）。生成的音频文件保存到服务器本地，路径和文本脚本一并更新到数据库的 `Song` 记录中，供播客模式调用。

管理员可以在后台触发"批量生成"任务，系统通过 `go func()` 以异步协程方式在后台执行，避免阻塞 HTTP 请求，并通过日志记录每批任务的执行进度和结果。

代码 5-10 管理员权限校验与系统重置
```go
func (*SystemAuth) ResetSystem(c *gin.Context) {
    user := GetCurrentUser(c)
    if user == nil || user.UserGroup != "admin" {
        ReturnError(c, g.ErrPermission, "权限不足，仅管理员可执行")
        return
    }

    dataDir := "data"
    absPath, err := filepath.Abs(dataDir)
    if err != nil {
        ReturnError(c, g.Err, "无法解析数据目录路径")
        return
    }

    files, err := filepath.Glob(filepath.Join(absPath, "*"))
    if err != nil {
        ReturnError(c, g.Err, "无法读取数据目录")
        return
    }

    deletedCount := 0
    for _, f := range files {
        baseName := filepath.Base(f)
        if strings.HasSuffix(baseName, ".db") || baseName == "data.db" || baseName == "music.db" {
            continue
        }
        if err := os.RemoveAll(f); err != nil {
            // 记录错误
        } else {
            deletedCount++
        }
    }

    db := GetDB(c)
    tx := db.Begin()
    tx.Exec("PRAGMA foreign_keys = OFF")
    tx.Exec("DELETE FROM playlist_songs")
    tx.Exec("DELETE FROM song_artists")
    tx.Exec("DELETE FROM songs")
    tx.Exec("DELETE FROM playlists")
    tx.Exec("DELETE FROM albums")
    tx.Exec("DELETE FROM artists")
    tx.Exec("DELETE FROM sqlite_sequence WHERE name IN ('songs','playlists','albums','artists')")
    tx.Where("user_group != ?", "admin").Delete(&model.User{})
    tx.Exec("PRAGMA foreign_keys = ON")

    if err := tx.Commit().Error; err != nil {
        tx.Rollback()
        ReturnError(c, g.ErrDbOp, "数据库提交失败")
        return
    }

    ReturnSuccess(c, gin.H{ "message": "系统重置成功", "deleted": deletedCount })
}
```

管理员可以进行以下操作：

- **用户管理**：查看所有注册用户的基本信息，包括用户名、邮箱、注册时间、最后登录 IP 和听歌时长；支持删除违规用户账号及修改用户权限分组。
- **公共歌单管理**：查看、创建、编辑和删除公共歌单；支持批量生成歌单的 AI 描述。
- **歌曲管理**：触发音乐库扫描任务，将本地音频文件自动导入系统数据库；查看所有歌曲的元数据；导出数据库到 Excel。
- **系统监控**：查看服务器实时状态，包括 CPU 占用率、内存占用率、Go 协程数量和 API 累计调用次数。

音乐扫描是后台的核心功能之一，由 `ScanUserMusic` 接口实现。后端遍历配置的音乐目录，递归读取 `.flac`、`.mp3`、`.wav` 等支持的音频文件，使用 `tag.ReadFrom` 解析文件的 ID3 元数据（标题、艺术家、专辑、年份、封面等），并调用原生音频解析工具获取时长、采样率、比特率等技术参数。扫描过程中，系统会自动根据文件所在文件夹名称创建对应歌单，将歌曲关联到歌单，并将封面图片缓存到本地。扫描完成后更新系统统计信息，包括总歌曲数、专辑数、艺术家数、总时长和总容量。

移动端一起听房间界面采用上下分栏布局，顶部为歌曲信息与播放控制，底部为聊天输入框与消息列表，成员列表通过标签页切换查看，移动端画面如图 5.22 所示。

后端 WebSocket 服务基于 `melody` 库和 Redis 实现，代码位于 `server/ws/server.go`。系统定义了多种消息类型，包括心跳、时间同步、聊天、加入/离开房间、播放控制（播放、暂停、跳转、切歌、变速）等。每个房间维护一条"权威时间线"（RoomTimeline），记录当前播放的歌曲 ID、开始播放的服务器时间戳、暂停状态和播放速度。如代码 5-11 所示。

代码 5-11 房间权威时间线与播放控制
```go
type RoomTimeline struct {
    SongID          string  `json:"song_id"`
    StartTimestamp  int64   `json:"start_timestamp"`
    Paused          bool    `json:"paused"`
    PausePositionMs float64 `json:"pause_position_ms"`
    Speed           float64 `json:"speed"`
}

func (t *RoomTimeline) ApplyPlay(serverTime int64) {
    t.StartTimestamp = serverTime - int64(t.PausePositionMs)
    t.Paused = false
}

func (t *RoomTimeline) ApplyPause(serverTime int64, positionMs float64) {
    t.PausePositionMs = positionMs
    t.Paused = true
}

func (t *RoomTimeline) ApplyChangeSong(serverTime int64, songId string) {
    t.SongID = songId
    t.StartTimestamp = serverTime
    t.PausePositionMs = 0
    t.Paused = false
}
```

房间成员信息和房主 ID 持久化存储在 Redis 的 Hash 和 String 数据结构中。当房主执行播放、暂停、切歌等操作时，后端首先校验操作者是否为房主，然后更新时间线状态并保存到 Redis，最后通过 `Broadcast` 将事件广播给所有在线连接的用户。前端收到事件后，根据服务器时间戳和本地计算的时间偏移量，将播放器状态同步到正确位置，从而实现多客户端的精准同步。

前端 WebSocket 客户端封装在 `src/core/realtime/socket.ts` 中，负责连接管理、心跳维持和时间同步。系统通过定期发送 `TIME_SYNC_REQ` 消息测量客户端与服务器之间的时间偏移量（Offset），用于校正播放进度，减少网络延迟带来的同步误差。聊天消息支持文本发送和成员列表实时更新，成员离开房间时系统会自动将房主身份移交给房间内其他在线用户。
