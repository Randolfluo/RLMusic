## 第2章 关键技术

### 2.1 开发环境和关键技术

本系统的开发与运行依赖于一系列成熟、高效的软硬件环境与工具链。

#### 2.1.1 硬件环境
*   **开发机器**：Windows 10/11 或 macOS，建议 16GB 以上内存（用于运行 Docker 及 Electron 调试）。
*   **服务器（可选）**：支持 Linux (Ubuntu/CentOS) 的 VPS 或本地 NAS，用于 Docker 容器化部署。

#### 2.1.2 软件环境
*   **操作系统**：Windows / macOS / Linux。
*   **编程语言**：Go 1.20+ (后端), Node.js 18+ (前端)。
*   **数据库**：SQLite 3 (默认嵌入式) 或 MySQL 5.7/8.0 (生产环境)。
*   **容器引擎**：Docker Engine & Docker Compose。

#### 2.1.3 开发工具
*   **IDE**：Visual Studio Code (前端/后端/Go/Vue 插件支持)。
*   **API 调试**：Postman 或 Apifox。
*   **版本控制**：Git。
*   **构建工具**：Vite (前端构建), Electron Builder (桌面打包), Capacitor CLI (移动端打包)。

### 2.2 关键技术

本系统采用前后端分离架构，融合了 Web、桌面与移动端开发技术，并集成了前沿的 AI 能力。

#### 2.2.1 前端核心技术 (Vue 3 + TypeScript)
*   **Vue 3 (Composition API)**：利用 Vue 3 的组合式 API 实现逻辑复用，相比 Options API 具有更好的类型推断支持和代码组织性，特别适合处理音乐播放器这种状态复杂的应用。
*   **TypeScript**：全项目采用 TypeScript 开发，提供静态类型检查，显著降低了跨模块调用（如 API 接口定义、Pinia 状态映射）时的运行时错误。
*   **Pinia & Persistence**：使用 Pinia 替代 Vuex 作为状态管理库，配合 `pinia-plugin-persistedstate` 插件，自动将播放进度、音量、用户 Token 等关键状态持久化到本地存储，防止刷新丢失。
*   **Naive UI**：采用 Naive UI 组件库构建现代化界面，其支持的主题定制能力便于实现深色/浅色模式切换。

#### 2.2.2 跨平台技术 (Electron & Capacitor)
*   **Electron**：用于构建 Windows/macOS 桌面客户端。通过主进程（Main Process）调用 Node.js API 访问本地文件系统（扫描音乐），并通过 IPC（进程间通信）与渲染进程交互，实现系统托盘、原生菜单等功能。
*   **Capacitor**：用于构建 Android 移动端应用。它提供了一层原生桥接（Native Bridge），使得 Vue 3 编写的 Web 页面能够在移动端 WebView 中运行，并能调用原生文件系统和音频能力。

#### 2.2.3 后端核心技术 (Go + Gin + Gorm)
*   **Go 语言**：选用 Go 语言作为后端开发语言，主要看重其极低的内存占用和强大的并发处理能力（Goroutine），非常适合在低功耗 NAS 或旧电脑上运行音乐服务。
*   **Gin 框架**：基于 Gin 构建 RESTful API，利用其高性能路由和中间件机制（如 JWT 鉴权、CORS 处理、日志记录）快速搭建 Web 服务。
*   **Gorm (ORM)**：使用 Gorm 作为对象关系映射框架，屏蔽了底层数据库（SQLite/MySQL）的差异，通过结构体（Struct）自动迁移数据库表结构（AutoMigrate），大幅降低了数据库维护成本。

#### 2.2.4 AI 内容增强技术
*   **LLM (大语言模型) 集成**：后端集成了阿里云 Qwen (通义千问) 或 SiliconFlow API。通过精心设计的 Prompt（提示词）模板，将歌手名、歌曲风格等元数据输入模型，自动生成高质量的歌手介绍或专辑文案。
*   **TTS (语音合成)**：利用 CosyVoice 等 TTS 模型，将生成的播客文案实时转换为自然流畅的语音文件（WAV/MP3），并在前端播放器中优先加载，实现“AI 电台”体验。

## 第3章 系统分析

### 3.1 可行性分析

在项目启动前，从技术、经济和操作三个维度进行了详细的可行性分析。

#### 3.1.1 技术可行性
*   **成熟的技术栈**：Vue 3 和 Go 均为当前业界主流且生态完善的技术，文档丰富，社区活跃。Electron 和 Capacitor 提供了成熟的跨端解决方案，无需维护多套代码库。
*   **性能保障**：Go 语言的高性能足以支撑家庭或中小规模团队的并发访问；SQLite 嵌入式数据库足以应对万级曲库的元数据存储，且无需额外部署数据库服务。
*   **AI 接口易用性**：各类大模型均提供了标准的 REST API，后端接入成本极低。
*   **结论**：技术上完全可行。

#### 3.1.2 经济可行性
*   **开发成本**：项目采用全开源技术栈，无软件授权费用。
*   **部署成本**：系统资源占用低，可部署在现有的闲置 PC、NAS 或低配云服务器上，无需购置昂贵的专用服务器。
*   **维护成本**：Docker 容器化部署使得升级和迁移变得极其简单，降低了运维人力成本。
*   **结论**：经济效益高，投入成本低。

#### 3.1.3 操作可行性
*   **用户友好**：前端界面参考主流流媒体软件设计，符合用户直觉，无需复杂培训即可上手。
*   **管理便捷**：提供可视化的 Web 管理后台，管理员可通过鼠标点击完成曲库扫描、用户管理等复杂操作，无需接触命令行。
*   **安装简单**：提供 Docker Compose 一键部署脚本和开箱即用的桌面端安装包，降低了用户的使用门槛。
*   **结论**：系统易于操作和推广。

### 3.2 需求分析

通过对目标用户（家庭用户、音乐爱好者）的调研，梳理出以下核心需求。

#### 3.2.1 功能需求

1.  **音乐资源管理**
    *   支持添加多个本地存储路径。
    *   支持自动扫描音频文件（MP3, FLAC, WAV 等）。
    *   支持解析 ID3 元数据（标题, 歌手, 专辑, 封面图）。
    *   支持封面图片的去重存储与展示。

2.  **播放控制**
    *   支持播放、暂停、上一首、下一首、进度拖拽。
    *   支持播放模式切换（顺序播放、随机播放、单曲循环）。
    *   支持实时歌词滚动显示。
    *   支持后台播放（移动端）和系统媒体中心控制（桌面端）。

3.  **用户与社交**
    *   支持用户注册、登录（JWT 鉴权）。
    *   支持创建、编辑、删除私有歌单。
    *   支持收藏歌曲（红心）、查看播放历史。
    *   支持“一起听”房间，多用户实时同步播放进度。

4.  **后台管理**
    *   支持管理员查看系统运行状态（CPU/内存/存储）。
    *   支持用户管理（封禁/解封/修改角色）。
    *   支持全库数据 Excel 导出备份。

5.  **AI 增强**
    *   支持生成歌手/专辑的智能介绍文案。
    *   支持生成并播放歌曲的“播客式”语音开场白。

#### 3.2.2 非功能需求

1.  **性能需求**
    *   曲库扫描速度应达到每秒处理 50+ 个文件（视磁盘 IO 而定）。
    *   Web 端首屏加载时间不超过 2 秒。
    *   播放器音频缓冲延迟低于 500ms。

2.  **兼容性需求**
    *   Web 端需兼容 Chrome, Firefox, Edge, Safari 等主流浏览器。
    *   桌面端需支持 Windows 10+ 和 macOS。
    *   移动端需适配 Android 10+ 设备。

3.  **安全性需求**
    *   API 接口需进行身份验证（Token）。
    *   用户密码需加密存储（Bcrypt）。
    *   防止越权访问（普通用户不能调用管理员接口）。
    *   导出文件需进行完整性自检，防止下载损坏文件。

4.  **可靠性需求**
    *   支持断点续传（音频流）。
    *   具备异常恢复能力（如扫描中断后可重新扫描）。
    *   Docker 容器具备自动重启策略。
